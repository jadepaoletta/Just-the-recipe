# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Commands

```bash
# Run both client and server in dev mode (recommended)
npm run dev

# Run individually
npm run dev:server   # Express server on :3001 (ts-node-dev, auto-restarts)
npm run dev:client   # Vite on :5173

# Type-check without emitting
cd server && npx tsc --noEmit
cd client && npx tsc --noEmit

# Lint (client only — ESLint configured)
cd client && npm run lint

# Build
npm run build

# Install all workspaces
npm run install:all
```

There are no tests in this project.

## Architecture

npm workspaces monorepo: `client/` (Vite + React + TypeScript) and `server/` (Express + TypeScript). They share no code — the client talks to the server exclusively via `fetch` calls to `/api/recipes`.

**Server** (`server/src/`):
- `index.ts` — Express setup, static file serving for `/data/images`, mounts the recipes router
- `routes/recipes.ts` — All API routes. The import route is the most complex: fetch page → scrape → Claude cleanup → insert recipe + ingredients + steps + download images
- `db.ts` — better-sqlite3 setup + schema creation. SQLite file lives at `data/recipes.db`. Migrations for new columns are done with a try/catch `ALTER TABLE` after the `CREATE TABLE IF NOT EXISTS` block
- `scraper.ts` — Fetches HTML with a plain HTTP client; falls back to Playwright (headless Chromium) if the fetch throws. Extracts recipe data via schema.org JSON-LD first, then CSS selector heuristics
- `claude.ts` — Anthropic SDK wrapper. Reads the API key from `process.env.CLAUDE_API_KEY`, falling back to parsing `secrets.sh` directly so the server works without sourcing it first. Currently exports `cleanScrapedRecipe`, which takes raw scraped data and returns a cleaned version with a generated 3–4 sentence description
- `imageDownloader.ts` — Downloads and resizes recipe images with sharp, stores them under `data/images/<recipeId>/`
- `units.ts` — US → metric unit conversion used when inserting ingredients

**Client** (`client/src/`):
- `api.ts` — All `fetch` calls to the server, typed against the interfaces in `types.ts`
- `pages/RecipeList.tsx` — Grid of recipe cards with search; opens `ImportModal`
- `pages/RecipeDetail.tsx` — Full recipe view with inline editing for title, ingredients, and steps; auto-saving notes textarea; image gallery
- `components/ImportModal.tsx` — URL input that POSTs to `/api/recipes/import`
- `index.css` — All styles. Uses CSS custom properties defined in `:root`. Fonts loaded from Google Fonts (Playfair Display for headings, DM Sans for body). No CSS modules or Tailwind.

**Data flow for recipe import:**
1. User pastes a URL → `POST /api/recipes/import`
2. Server fetches the page HTML (Playwright fallback for JS-rendered sites)
3. `scrapeRecipe` extracts title, description, ingredients, steps, image URLs
4. `cleanScrapedRecipe` sends the raw scraped data to Claude (Haiku) and gets back a cleaned version with a generated description; sets `description_ai_generated = 1` in the DB
5. Recipe, ingredients, and steps are inserted into SQLite; images are downloaded in parallel

**Persistent data** lives outside source control in `data/` (gitignored):
- `data/recipes.db` — SQLite database
- `data/images/<recipeId>/` — Downloaded recipe images

**API key** is stored in `secrets.sh` at the project root (gitignored). The server reads it directly from that file as a fallback — no need to `source secrets.sh` before starting the dev server.

**AI badge**: recipes where the description was generated by Claude have `description_ai_generated = 1` in the DB. The detail page renders a small amber pill badge next to the description when this flag is set.
